We can model the ship as a torsion pendulum. The idea is that we want the pendulum
to have a critically damped oscillation towards a target angle. The moment of inertia, angular
position/velocity/acceleration are all fully known. We can use an applied torque to efficiently
dissipate the rotational energy such that the ship points to the target.

https://farside.ph.utexas.edu/teaching/301/lectures/node139.html <== torsion pendulum model

So if we model our situation like this, the force equation is

T (torque) = - k (torsion constant) * x (angular position) - D (damping) * angvel (angular velocity)

which when converted to a differential equation, noting that

w^2 = k/I and h = D/I

we write

d^2/dt^2 x = -w^2 * x - h * d/dt x

which has a solution like a damped oscillator,

x(t) = Ae^((-h/2)x) * cos(wt-phi (phase))

Plotting this with arbitrary A and phi values reveals that it is indeed the correct motion.
We look at the damping ratio (I'm gonna say zeta = Z)

Z = D / (2I*w)
  = D / (2I*sqrt(K/I))
  = D / (2*sqrt(KI))		<== we pushed the I into the sqrt and canceled the 1/I

and note that if Z = 1, then we have critical damping. So, if Z = 1,

1 = D / (2*sqrt(KI))
D = 2*sqrt(KI)

So it seems that we can just make the damping force 2*sqrt(KI). If we make K higher,
then I bet the system will damp out faster; however if its too high, we'll get
errors due to the time discretization.

Targeting system update idea! The guidance law I've used so far is known as "pursuit" navigation. It's
a dumb guidance law that works by simply aiming at the taget and going forward. This creates the 
missile-circles-around-target problem. To fix this, I can try to implement proportional navigation.
To do this, I need to steer the missile such that the bearing between the target and the missile
is constant. If the bearing is constant, then the missile is on a collision course.

https://en.wikipedia.org/wiki/Proportional_navigation

I can use the equation

a_n = N * lambda-dot * V

where N is an integer between 3 and 5, inclusive; lambda-dot is the rate of change of the Line-of-Sight (LOS)
angle; V is the magnitude rate of change of the distance between the target and the missile. a_n is the angular
acceleration of the missile.

To implement this, I need a way to find the rate of change of the LOS angle in 2D. I could do a simple numerical
derivative. I could use a buffer array to store the LOS angle for 2 timesteps. It would look like

[1.2, 1.5]

Then I find the average rate of change through division,

lambda-dot = (1.2 + 1.5) / (time2 - time1)

For the difference in time, I can use dt. This is a first-order error backwards derivative. For my game,
this seems to lead to errors. To correct this, I can use a second-order error centered derivative. Let's say
our buffer array looks like this:

B1: [x_t+1, x_t, x_t-1] (first in, first out, where x_t-1 occurs at t-1, an earlier time)

lamdba-dot = (x_t+1 - x_t-1)/(2dt)

This way, I can reduce errors quickly by just making dt a little smaller. To reduce errors for a first order method,
I would need much, much smaller dt values.

If the change in angle is positive, then the LOS angle is moving counter-clockwise (think unit circle). If it's
negative, then the LOS angle is moving clockwise. So I don't need to change the sign of the angular acceleration.

The angle buffer array could be stored with each missile. It should be a 1D float array with 2 elements.

To update the buffer array, I would need a method to push elements through an array like so:

initial array [1,2] --> adding element [1,2,3] --> returning trimmed array [2,3]. I bet c++ has a method for this.

Some light research from 
https://stackoverflow.com/questions/45503324/add-an-element-at-the-beginning-of-an-array-and-shift-the-rest-but-keep-the-size
tells me that I can use pop_back(). It does change the size of the buffer array temporarily, but I doubt that will
slow this down that much.

The physics engine turns objects by applying a torque, and since proportional navigation uses angular acceleration, I can use
T = I*a_n. I also have a way to calculate the moment of inertia for objects.

== UPDATING ANGLE DIFF FUNCTION ==

I notice an issue that's impacting my physics engine in general. Consider the hour hand on a clock.

I have stuff that would find the angular distance the hand is from another point on the clock. Now, on a clock,
that distance typically involves only one direction of thinking depending on context. If the hour hand was at 12 am, and I
asked "how far it is from 3 am", then I see the short distance between 12 am and 3 am, and answer "three hours".

This ignores that there are two distances; the major and minor arcs that rotating systems involve! I have an expression that
finds the difference in the angles of two objects, but it breaks when the two angles are on different sides of the polar axis.
What I need to do instead is calculate which direction is smaller, and have my systems use that angle instead.

So to fix this: if the difference in angles is greater than 180 degrees, then the result I care about is actually
360 - the difference in angles.

pseudocode:

angle_diff = angle1 - angle2
if angle_diff < 180:
	return angle_diff
else:
	return 360 - angle_diff
